---
title: "Project structure: Oiyo's directory conventions explained"
description: "Oiyo uses a conventional directory layout in src/ to enable auto-routing, layout resolution, and page meta—without any manual registration."
---

Oiyo's features—auto-routing, the layout system, root context, and page meta—are all powered by knowing where to look. Rather than requiring explicit configuration to locate pages or layouts, Oiyo scans well-known directories at build time and wires everything together automatically. Understanding the directory layout helps you navigate the codebase, predict how changes propagate, and avoid the few places where manual edits would be overwritten.

## Directory layout

```
my-app/
├── src/
│   ├── App.vue          # App shell, root context, layout host
│   ├── main.ts          # App entry point using createSSRApp
│   ├── manifest.json    # UniApp app manifest
│   ├── pages.json       # Auto-generated page registry (managed by Oiyo)
│   ├── static/          # Static assets
│   ├── layouts/         # Named layout components
│   │   ├── default.vue  # Default layout for pages without explicit layout
│   │   └── tabbar.vue   # Tab bar layout for main entry pages
│   └── pages/           # File-system based pages
│       ├── home/
│       │   └── index.vue
│       └── my/
│           └── index.vue
├── vite.config.ts       # Vite config with OiyoPlugin
├── uno.config.ts        # UnoCSS config
├── tsconfig.json
└── package.json
```

<AccordionGroup>
  <Accordion title="src/App.vue — app shell, root context, and layout host">
    `App.vue` serves three distinct roles in an Oiyo project.

    **App shell.** It is the outermost component rendered on every page transition. Markup placed directly in its template appears on all screens—useful for a global background, a status bar overlay, or an analytics tracker.

    **Root context.** `defineRootContext` is called in `App.vue`'s `<script setup>` to create reactive state that is accessible in every page and layout without props or a store. The returned refs and functions are both local to `App.vue` and injectable anywhere downstream.

    ```vue src/App.vue
    <script setup lang="ts">
    import { onLaunch, ref } from '@skiyee/oiyo/runtime'

    const { title, count, increment } = defineRootContext(() => {
      const title = ref('Hello Oiyo')
      const count = ref(2)

      function increment() {
        count.value += 3
      }

      return { title, count, increment }
    })

    onLaunch(() => {
      console.log('App Launch, title:', title.value)
    })
    </script>
    ```

    **Layout host.** The `<OiyoLayout>` component in `App.vue`'s template is the mount point for the layout system. Oiyo reads the current page's `definePageMeta` declaration, resolves the named layout component from `src/layouts/`, and renders it inside `<OiyoLayout>`. `<OiyoPage>` inside the layout renders the page's own template.

    ```vue src/App.vue (template excerpt)
    <template>
      <view>
        <!-- App-level chrome -->
        <OiyoLayout>
          <OiyoPage />
        </OiyoLayout>
      </view>
    </template>
    ```
  </Accordion>

  <Accordion title="src/main.ts — app entry point">
    `main.ts` exports a `createApp` factory function required by UniApp's SSR runtime. Oiyo re-exports `createSSRApp` from its runtime package so you import from a single location.

    ```typescript src/main.ts
    import { createSSRApp } from '@skiyee/oiyo/runtime'
    import App from './App.vue'
    import 'uno.css'

    export function createApp() {
      const app = createSSRApp(App)
      return {
        app,
      }
    }
    ```

    The `uno.css` import activates UnoCSS styles globally. If you are not using UnoCSS, remove this line and the corresponding `UnoPlugin` from `vite.config.ts`.

    <Note>
      UniApp requires the SSR-compatible `createSSRApp` on all platforms—including H5 and mini-programs. Do not substitute `createApp` from Vue core.
    </Note>
  </Accordion>

  <Accordion title="src/pages.json — auto-generated page registry">
    `pages.json` is the UniApp page manifest. Oiyo generates and maintains this file entirely. You should never edit it by hand.

    ```json src/pages.json
    {
      "pages": [
        {
          "path": "pages/home/index",
          "layout": "tabbar",
          "style": {
            "navigationBarTitleText": "首页 - Oiyo"
          }
        },
        {
          "path": "pages/my/index",
          "style": {
            "navigationBarTitleText": "我的 - Oiyo"
          }
        }
      ]
    }
    ```

    The data in this file comes from each page's `definePageMeta` call. When you run `oiyo prepare` (or `pnpm install`, which triggers it via `postinstall`), Oiyo scans `src/pages/`, reads each file's meta declaration, and writes the corresponding entries here.

    <Warning>
      Any manual changes to `pages.json` will be overwritten the next time `oiyo prepare` runs. Manage page configuration exclusively through `definePageMeta` in each page file.
    </Warning>
  </Accordion>

  <Accordion title="src/layouts/ — named layout components">
    Every `.vue` file in `src/layouts/` is a named layout. The filename (without extension) is the layout identifier used in `definePageMeta`.

    **`default.vue`** is the fallback layout. Any page that does not declare an explicit `layout` in its `definePageMeta` is automatically wrapped in `default.vue`.

    ```vue src/layouts/default.vue
    <template>
      <view>
        <!-- Default layout chrome -->
        <slot />
      </view>
    </template>
    ```

    **`tabbar.vue`** is a named layout intended for main entry pages that carry tab bar navigation semantics. A page opts in with `definePageMeta({ layout: 'tabbar' })`.

    ```vue src/layouts/tabbar.vue
    <template>
      <view>
        <!-- Tab bar layout chrome -->
        <slot />
      </view>
    </template>
    ```

    The `<slot />` in each layout is where Oiyo renders the page's own template content. You can add persistent chrome (headers, footers, nav bars) around the slot.

    <Tip>
      You can add as many named layouts as your project needs. Create `src/layouts/auth.vue` and reference it with `definePageMeta({ layout: 'auth' })` in any page that needs the authentication chrome.
    </Tip>
  </Accordion>

  <Accordion title="src/pages/ — file-system based pages">
    Every `.vue` file nested inside `src/pages/` becomes a route. The route path mirrors the file path relative to `src/`.

    | File | Route path |
    | ---- | ---------- |
    | `src/pages/home/index.vue` | `/pages/home/index` |
    | `src/pages/my/index.vue` | `/pages/my/index` |

    Each page declares its own configuration using the `definePageMeta` compiler macro:

    ```vue src/pages/home/index.vue
    <script setup>
    import { onLoad, ref } from '@skiyee/oiyo/runtime'

    definePageMeta({
      layout: 'tabbar',
      style: {
        navigationBarTitleText: '首页 - Oiyo',
      },
    })

    onLoad(() => {
      console.log('Page Load')
    })

    const title = ref('Hello')
    </script>
    ```

    Pages that omit `layout` fall back to `default.vue`:

    ```vue src/pages/my/index.vue
    <script setup>
    definePageMeta({
      style: {
        navigationBarTitleText: '我的 - Oiyo',
      },
    })
    </script>
    ```

    `definePageMeta` is a compile-time macro. It does not execute at runtime and has zero bundle cost. All values must be statically analyzable literals—you cannot pass dynamic expressions or variables.
  </Accordion>

  <Accordion title="vite.config.ts — Vite configuration">
    All of Oiyo's build-time features—auto-routing, `definePageMeta` extraction, layout resolution—are implemented as Vite plugin transforms registered through `OiyoPlugin`.

    ```typescript vite.config.ts
    import OiyoPlugin from '@skiyee/oiyo/plugin'
    import UnoPlugin from 'uni-unocss'
    import { defineConfig } from 'vite'

    export default defineConfig({
      plugins: [
        UnoPlugin(),
        OiyoPlugin(),
      ],
    })
    ```

    `UnoPlugin` must be listed before `OiyoPlugin` so that UnoCSS class scanning completes before Oiyo's transforms run. No other Oiyo-specific configuration is required in `vite.config.ts`—all framework behavior is configured through `definePageMeta` and the directory conventions.
  </Accordion>

  <Accordion title="uno.config.ts — UnoCSS configuration">
    Oiyo's example project uses UnoCSS with the `presetUni` preset, which provides UniApp-compatible utility classes (using `rpx` units, mini-program compatible property names, etc.).

    ```typescript uno.config.ts
    import {
      defineConfig,
      presetUni,
      transformerDirectives,
      transformerVariantGroup,
    } from 'uni-unocss'

    export default defineConfig({
      presets: [
        presetUni(),
      ],
      transformers: [
        transformerDirectives(),
        transformerVariantGroup(),
      ],
    })
    ```

    `transformerDirectives` enables `@apply` in `.vue` files. `transformerVariantGroup` enables the `hover:(bg-red text-white)` shorthand syntax. Both are optional.
  </Accordion>
</AccordionGroup>

## Naming conventions

<AccordionGroup>
  <Accordion title="Layout auto-discovery">
    Layouts are discovered by filename. Any `.vue` file placed directly inside `src/layouts/` is registered automatically with the name equal to its filename (without `.vue`). There is no index file, no explicit import, and no registration step.

    - `src/layouts/default.vue` → layout name `"default"`
    - `src/layouts/tabbar.vue` → layout name `"tabbar"`
    - `src/layouts/auth.vue` → layout name `"auth"`

    Subdirectories inside `src/layouts/` are not scanned. Keep all layout files at the top level of the directory.
  </Accordion>

  <Accordion title="Page auto-discovery">
    Pages are discovered recursively from `src/pages/`. Any `.vue` file at any nesting depth is treated as a page. The file's path relative to `src/` becomes the route path.

    The conventional pattern is to nest each page in a directory named after the route segment, with the file named `index.vue`. This matches how UniApp internally resolves routes and keeps the pages directory scannable at a glance.

    - `src/pages/home/index.vue` → route `/pages/home/index`
    - `src/pages/profile/settings/index.vue` → route `/pages/profile/settings/index`
  </Accordion>

  <Accordion title="pages.json is owned by Oiyo">
    `src/pages.json` is generated output, not source. Treat it like a compiled artifact: commit it to version control (UniApp needs it), but never edit it directly. The source of truth for every entry in `pages.json` is the corresponding `definePageMeta` call in the page file.

    To regenerate `pages.json` at any time without running the dev server, run:

    ```bash
    oiyo prepare
    ```

    This is also the command that `postinstall` runs automatically after `pnpm install`.
  </Accordion>
</AccordionGroup>
